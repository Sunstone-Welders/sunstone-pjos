# Tap to Pay — native build setup

This doc covers the one-time work needed on the Mac / Apple Developer Console /
Square Dashboard side after the cross-platform Tap to Pay code lands. The TS
side is automated; the native side is not.

## Architecture: split init vs. activate

The native plugin exposes two separate entry points so the artist never sees
Square's settings sheet during normal cold-start:

| Method | When | UI? |
|---|---|---|
| `SquareTapToPay.initialize(...)` | Dashboard mount | None — silent |
| `SquareTapToPay.activateReader()` | POS / Event Mode mount | Sunstone branded overlay, auto-dismisses Square's settings sheet on reader-connect |

`initializeTapToPay()` in `src/lib/tap-to-pay.ts` handles authorization only
and is deduped per process. `activateTapToPayReader()` is also deduped per
process — a successful activation short-circuits subsequent calls.

The `TapToPayActivationGate` wrapper on the POS and Event Mode pages decides
whether to mount the `TapToPayActivation` overlay based on:

1. Capacitor native platform (iOS)
2. `SquareTapToPay.isAvailable()` (device + OS supports Tap to Pay)
3. `SquareTapToPay.hasProximityReaderEntitlement()` (Apple's
   `com.apple.developer.proximity-reader.payment.acceptance` is in the
   running binary's entitlements)
4. Tenant has `tap_to_pay_enabled = true` AND `square_merchant_id` set

Production-signed builds without the entitlement hide every Tap to Pay
surface end-to-end because (3) fails.

Plugin events surfaced to JS:

- `readerConnected` — `{ model: "tapToPay" }` — fires whenever the embedded
  Tap to Pay reader attaches. Overlay subscribes to this so it dismisses even
  when activation completes via a different code path.
- `readerActivationTimedOut` — fires after the 45s safety timeout if the
  reader never attaches.


## 0. Environment

Server-side env var (already used by `/api/square/mobile-payments-auth`):

```
SQUARE_APPLICATION_ID=sq0idp-...     # platform's Square dev app ID
```

This is the **Square developer application ID** (from the Square Dashboard),
not the merchant ID, and not the Android app ID. The Android SDK needs it
for `MobilePaymentsSdk.initialize(applicationId, application)`. iOS ignores
this value.

Set it in Vercel (Production + Preview) before deploying the Settings UI.

## 1. iOS — one-time Xcode setup

The Square Mobile Payments SDK is added via Swift Package Manager. **Do not
edit `ios/App/CapApp-SPM/Package.swift`** — it's regenerated by `cap sync`.
Add the dependency to the App target's project itself:

1. Open `ios/App/App.xcworkspace` in Xcode.
2. Project navigator → select the **App** project → **Package Dependencies** tab.
3. Click **+**, paste `https://github.com/square/mobile-payments-sdk-ios` and
   choose **Up to Next Major Version, starting from 2.5.0** (or pin to the
   latest 2.x).
4. When the package resolves, in the "Add to Target" dialog select the **App**
   target and add the `SquareMobilePaymentsSDK` product.
5. Project navigator → **App** target → **Build Phases** → **Compile Sources**.
   Verify `App/Plugins/SquareTapToPay/SquareTapToPayPlugin.swift` and
   `SquareTapToPayPlugin.m` are listed. If not, drag the folder in.

### Tap to Pay on iPhone entitlement

Apple gates this capability. Before a signed build will succeed:

1. In the Apple Developer Console, request the **Tap to Pay on iPhone**
   capability for the Sunstone team account (Apple-reviewed; can take days).
2. Once granted, edit the App ID `com.sunstoneengineering.studio`, enable
   **Tap to Pay on iPhone**, and re-generate the provisioning profile.
3. Pull the new profile in Xcode (Signing & Capabilities → Automatic).

The entitlement `com.apple.developer.proximity-reader.payment.acceptance` is
already declared in `ios/App/App/App.entitlements`.

### Info.plist usage strings

Already added in this PR:
- `NSBluetoothAlwaysUsageDescription` — for optional Square readers.
- `NSLocationWhenInUseUsageDescription` — required by Square for sale
  verification.
- `NSMicrophoneUsageDescription` — for the optional magstripe reader.

### Device requirements

- iPhone XS or newer (Tap to Pay on iPhone hardware requirement).
- iOS 16.4+.

## 2. Android — one-time setup

Already wired in this PR:
- `android/variables.gradle` — `minSdkVersion` bumped 24 → 28.
- `android/app/build.gradle` — `com.squareup.sdk:mobile-payments-sdk:2.4.0`.
- `AndroidManifest.xml` — NFC + Bluetooth + Location + Microphone permissions.
- `MainActivity.java` — registers `SquareTapToPayPlugin`.

After pulling: `npx cap sync android && npx cap open android`. The build should
resolve `com.squareup.sdk:mobile-payments-sdk` from mavenCentral.

### Device requirements

- Android 9 / API 28+.
- NFC-capable hardware (most phones since 2018).

## 3. Square Dashboard

In Square's developer dashboard, on the platform application:

1. Confirm the application is in **Production** mode for live sales (Sandbox
   tokens won't authorize Tap to Pay on a real card).
2. The OAuth scopes for the artist's connection must include
   `PAYMENTS_WRITE`, `MERCHANT_PROFILE_READ` — already requested by the
   existing `/api/square/authorize` flow.

## 4. Verification flow (real device)

1. Deploy the Vercel build that includes `tap-to-pay.ts` + Settings UI.
2. On the device: open the app, go to **Settings → Payments → Tap to Pay on
   this device** and tap **Set Up Tap to Pay**.
   - iOS: the Square SDK presents Apple's Tap to Pay terms — accept them.
   - Android: no T&C step.
3. Toast should read "Tap to Pay is ready."
4. **Cold-launch activation**: force-quit the app, relaunch, land on the
   dashboard. No reader UI should appear. Navigate to **POS** — the branded
   "Setting up your card reader" overlay should appear, Square's settings
   sheet should pop up on top, and as soon as the embedded reader attaches
   the entire flow auto-dismisses. POS is visible underneath the whole time.
5. **Within-session re-navigation**: after step 4 succeeds, navigate away
   from POS and back. No activation overlay should appear — the reader is
   already attached for this process.
6. Add an item, hit checkout. The Tap to Pay button is the first option on
   the payment screen.
7. Tap it → Square's payment sheet appears → hold a real card to the phone.
8. After success, the sale row in `sales` should have:
   - `payment_method = 'square_tap'`
   - `payment_provider = 'square'`
   - `payment_provider_id` = Square payment ID
   - `payment_status = 'completed'`

## 5. Troubleshooting

- **iOS build fails with "missing entitlement"**: Apple hasn't granted the
  Tap to Pay capability yet, or the provisioning profile hasn't been
  regenerated. See section 1.
- **"applicationId is required on Android"** error in the app: `SQUARE_APPLICATION_ID`
  isn't set in Vercel's env. Add it and redeploy.
- **"Square authorization failed"**: the artist's `square_access_token` is
  stale. Have them reconnect Square from Settings.
